In tokenization, we learned that the model does not directly read words like humans do.

"I love AI"

Tokens:
["I", "love", "AI"]

Token IDs:
[40, 3021, 15836]

But token IDs are only numbers for lookup. They do not carry meaning by themselves. So the next question is: how does the model convert token IDs into meaningful mathematical form?

That answer is: Embeddings.

2. What Is an Embedding?

An embedding is a vector representation of meaning. It converts a token, word, sentence, paragraph, image, or document into a list of numbers that captures meaning. 

"king" → [0.21, -0.45, 0.88, 0.12, ...]

In real LLMs, the vector may have hundreds or thousands of numbers.

king  → [0.21, -0.45, 0.88, 0.12, ...]
queen → [0.19, -0.41, 0.84, 0.18, ...]
car   → [-0.72, 0.33, -0.15, 0.49, ...]

The idea is:

king and queen are close in meaning
king and car are far in meaning
Embedding = meaning converted into numbers.

3. Why Token IDs Are Not Enough

Suppose we have:

WordID
king101
queen102
apple103
car104

These numbers do not tell the model that:

king is closer to queen
apple is closer to fruit
car is closer to vehicle

Token IDs are like database primary keys. They identify the token, but they do not explain meaning. The ID 101 does not mean royal, male, palace, crown, or kingdom.

So we need another representation. That is the embedding.

4. Simple Mental Model

Think of embedding as placing words in a meaning-space.

             queen
              |
              |
king -------- palace -------- crown


apple ------- fruit -------- mango


car -------- vehicle ------- bus

Words with related meaning are placed near each other. Words with unrelated meaning are placed far away. In actual LLMs, this space is not 2D or 3D. It may be 768, 1536, 3072, 4096 dimensions or more.

5. Embedding as Coordinates

Suppose we create a very simple 2D embedding space. Let dimension 1 be royalty meaning and dimension 2 be food meaning.

WordRoyaltyFood
king0.950.05
queen0.920.04
prince0.880.03
apple0.020.95
mango0.010.92
banana0.010.90
car0.050.02

Now we can see:

king ≈ queen ≈ prince
apple ≈ mango ≈ banana
car is not close to either group
This is the core idea: meaning can be represented as coordinates.

    6. Important Point: Embeddings Are Learned

    Embeddings are not manually written by humans.

    Nobody sits and writes:
king  = [0.95, 0.10]
queen = [0.92, 0.12]
apple = [0.05, 0.90]

    During training, the model learns that words used in similar contexts should have similar vectors. 

    The king ruled the kingdom.
The queen ruled the kingdom.
The prince lived in the palace.

    The model sees that king, queen, and prince appear in similar contexts. So their vectors become close.

    I ate an apple.
I ate a mango.
I ate a banana.

    Similarly, apple, mango, and banana become close because they appear in similar contexts.

    Words used in similar contexts should have similar vectors.

    7. From Token ID to Embedding

    Inside the model, there is usually an embedding table.

    TokenToken IDEmbedding Vector

    When the input is:

    king queen apple

    Tokenization gives:

    [101, 102, 105]

    Then embedding lookup gives:

    101 → [0.95, 0.10]
102 → [0.92, 0.12]
105 → [0.05, 0.90]

    So the model no longer works with raw token IDs. It works with vectors.

    8. Embedding Lookup Is Like a Dictionary

    A simple analogy is a JavaScript dictionary:

    embeddingTable = {
  101: [0.95, 0.10],
  102: [0.92, 0.04],
  103: [0.02, 0.95],
  104: [0.05, 0.02]
}

tokenId = 101
output = [0.95, 0.10]

    This is the first major mathematical transformation inside the LLM.

    Token ID → Embedding Vector

    9. What Does Each Number Mean?

    In simple teaching examples, we may say dimension 1 means royalty, dimension 2 means food, and dimension 3 means vehicle. But in real LLMs, dimensions are not so clean. 

    king = [0.21, -0.45, 0.88, 0.12, 0.66, ...]

    The meaning of “king” is spread across the full vector. This is called distributed representation.

    A single dimension may not mean exactly one human concept. Meaning is distributed across many numbers.

    10. Similarity Between Embeddings

    Once words or sentences become vectors, we can compare them mathematically. A common method is cosine similarity. It checks whether two vectors point in a similar direction.

    PairSimilarityMeaning
    similarity(king, queen)0.91Very similar
    similarity(king, apple)0.18Weak relation
    similarity(apple, mango)0.88Similar
    Cosine SimilarityMeaning
    0.90+Very similar
    0.70–0.89Related
    0.40–0.69Some relation
    0.00–0.39Weak relation
    NegativeOpposite or unrelated direction

    11. Tiny Cosine Similarity Intuition

    Suppose:

    king  = [0.95, 0.05]
queen = [0.92, 0.04]
apple = [0.02, 0.95]

    king and queen point almost in the same direction. So similarity is high. king and apple point in very different directions. So similarity is low.

    This is why embeddings are useful for:

    search
recommendation
clustering
RAG
semantic matching
    Embeddings make meaning measurable.

    13. Why Embeddings Are Powerful

    Embeddings move us from exact word matching to meaning matching.

    Old Search

    Query: car insurance

    Old keyword search may only find documents containing exactly:

    car insurance

    Embedding-Based Search

    Embedding search can also find:

    vehicle policy
auto coverage
motor claim protection

    because the meaning is close.

    This is the heart of semantic search: search by meaning, not only by exact words.

    14. Embeddings vs One-Hot Encoding

    Suppose vocabulary has five words:

    king, queen, apple, mango, car
    WordOne-Hot Vector
    king[1, 0, 0, 0, 0]
    queen[0, 1, 0, 0, 0]
    apple[0, 0, 1, 0, 0]
    mango[0, 0, 0, 1, 0]
    car[0, 0, 0, 0, 1]

    Problem:

    king and queen look completely unrelated.
apple and mango look completely unrelated.

    Embedding example:

    WordEmbedding Vector
    king[0.95, 0.05]
    queen[0.92, 0.04]
    apple[0.02, 0.95]
    mango[0.03, 0.91]
    car[0.50, 0.20]
    In one-hot encoding, all words are equally different. Embeddings solve this.

    15. Document Embeddings and Chunks

    When we build RAG or semantic search, we usually do not embed the full book or full PDF as one big vector. Instead, we break it into smaller parts called chunks.

    Why chunks?

    Page 1: Leave policy
Page 2: Refund policy
Page 3: Security policy
Page 4: Travel policy

    If we convert the full document into one embedding, the meaning becomes too mixed. Better approach: 

    Document
 ↓
Split into chunks
 ↓
Embed each chunk separately
 ↓
Store each chunk vector
    Chunk IDChunk Text
    C1Customers can request a refund within 14 days of purchase.
    C2Employees can apply for annual leave through the HR portal.
    C3All users must enable multi-factor authentication.
    C4Business travel must be approved by the manager.

    16. Embeddings in RAG

    RAG means Retrieval-Augmented Generation.

    User Question
 ↓
Convert question to embedding
 ↓
Search similar document embeddings
 ↓
Retrieve relevant chunks
 ↓
Send chunks + question to LLM
 ↓
Generate answer

    Example:

    Question:
What is our refund policy?

    Embedding search may find a document chunk like:

    Customers can request reimbursement within 14 days...

    Even if the exact word “refund” is not used.

    That is why embeddings matter so much in enterprise AI.

    19. Keyword Search vs Embedding Search

    Keyword search looks for exact or near-exact words. Embedding search looks for meaning.

    User QueryDocument TextKeyword SearchEmbedding Search
    money backrefund policyWeakStrong
    forgot loginreset passwordWeak / MediumStrong
    sick leavemedical absenceWeakStrong
    car insurancevehicle coverageWeakStrong
    price cutdiscountWeakStrong
    laptop not startingdevice power issueWeakStrong

    But keyword search is not useless. It is very useful for exact things:

    invoice number INV-10291
employee ID E4582
error code ORA-12514
order ID ORD-7781
function name CalculateTax()
table name CustomerOrders
    Best real-world approach: Hybrid Search = keyword search + embedding search.

    20. How Embeddings Are Learned During Training

    At the beginning of training, the embedding table starts with random values.

    TokenToken IDInitial Random Embedding
    king101[0.13, -0.52]
    queen102[-0.44, 0.31]
    apple103[0.77, -0.10]

    Then the model sees text again and again:

    The king lives in the palace.
The queen lives in the palace.
The prince lives in the palace.

I ate an apple.
I ate a mango.
I ate a banana.

    How does the model know it is wrong?

    The king lives in the ______

Expected answer:
palace

If the model predicts:
banana

That is wrong. Loss is high.
Training adjusts internal numbers, including embeddings.

    In neural network language, embedding values are trainable parameters. Like linear regression adjusted m and c, LLMs adjust embedding weights, attention weights, feed-forward weights, and many other parameters.

    Meaning is learned from how words appear with other words.

    22. Positional Embeddings

    Embeddings give meaning to tokens. But the model also needs to know position and order.

    Ram loves Sita.
Sita loves Ram.

    Both sentences have the same words, but the meaning changes because order changes.

    InputVector(token) = TokenEmbedding(token) + PositionEmbedding(position)

    Sentence: Ram loves Sita

    TokenPositionToken EmbeddingPosition EmbeddingFinal Vector
    Ram1[0.80, 0.10][0.01, 0.03][0.81, 0.13]
    loves2[0.40, 0.60][0.02, 0.04][0.42, 0.64]
    Sita3[0.78, 0.12][0.03, 0.05][0.81, 0.17]

    Sentence: Sita loves Ram

    TokenPositionToken EmbeddingPosition EmbeddingFinal Vector
    Sita1[0.78, 0.12][0.01, 0.03][0.79, 0.15]
    loves2[0.40, 0.60][0.02, 0.04][0.42, 0.64]
    Ram3[0.80, 0.10][0.03, 0.05][0.83, 0.15]

    Same token, different position, different final input vector.

    Token embeddings tell what the word means. Positional embeddings tell where the word is.

    23. Final Summary

    Embeddings allow AI systems to search, compare, and retrieve knowledge by meaning instead of exact words. 
    Text → Tokens → Token IDs → Embeddings → Transformer / Vector Search → RAG / Answer
    Token ID identifies the token.
Embedding vector represents meaning.
Similar meanings have similar vectors.
Embeddings are learned from context.
Embedding lookup is like a dictionary.
Distance measures how far meaning points are.
Cosine similarity compares meaning direction.
Word embeddings, sentence embeddings, and document embeddings are used differently.
Embeddings make semantic search powerful.
Document chunks + embeddings support RAG.
Vector databases store and search embedding vectors.
Positional embeddings add order information.
